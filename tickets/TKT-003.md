## Description

The posts rendering layer (src/pages/posts/[slug].astro) currently has no navigation between posts. When reading a post, there's no way to go to the next or previous post without going back to the listing.

Add prev/next links at the bottom of each post page. Follow the pattern used by src/layouts/Longform.astro for deep-dive sibling navigation (queries collection, sorts, finds current entry, renders prev/next links), but keep it simpler for posts.

Implementation approach:
- In src/pages/posts/[slug].astro, query the posts collection, sort by date descending
- Find the current post's position in the sorted list
- Pass prevPost and nextPost to the layout (or render nav directly after `<RenderedContent />`)
- Render a simple prev/next bar at the bottom: left arrow + previous post title on the left, next post title + right arrow on the right
- Style using existing CSS custom properties (var(--color-text-muted), var(--color-border), etc.)

This is a framework file, so it must work generically for any Loomwork site with posts. No site-specific hardcoding.


## Acceptance Criteria

- [ ] Post pages display prev/next navigation links at the bottom
- [ ] "Previous" links to the older post (earlier date), "Next" links to the newer post (later date)
- [ ] First post (oldest) has no prev link; most recent post has no next link
- [ ] Navigation styling matches existing site patterns using CSS custom properties
- [ ] No em dashes in any copy
- [ ] No site-specific hardcoding; works generically for any Loomwork site with posts
- [ ] npm run build passes

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### Current state

`src/pages/posts/[slug].astro` (lines 1-74) is a dynamic route that:
1. Fetches all non-draft posts via `getCollection("posts")` in `getStaticPaths()` (line 6)
2. Receives the single `post` via `Astro.props` (line 13)
3. Renders it inside `Content.astro` layout with post meta (author, tags) and `<RenderedContent />` (lines 17-38)
4. Has scoped CSS for `.post-meta`, `.post-meta__author`, `.post-meta__tags`, `.post-meta__tag` (lines 40-74)

There is **no prev/next navigation** of any kind. After reading a post, the user must go back to `/posts/` to find another.

### Existing pattern to follow

`src/layouts/Longform.astro` (lines 46-59) demonstrates sibling navigation for longform pages:
- Fetches the collection, filters by section and template
- Maps entries to `{ title, slug, nav_order }`, sorts by `nav_order`
- Renders a sidebar listing all siblings with the current page highlighted

For posts, the approach is simpler: sort by `date` descending, find current index, derive prev/next.

### Content schema

`src/content.config.ts` (lines 53-66) defines the posts collection with required fields: `title`, `description`, `date` (coerced Date). The `date` field is what we sort by. Post slugs are derived from `post.id.replace(/\.mdx?$/, "")` (seen at line 9 of `[slug].astro` and line 26 of `posts/index.astro`).

### Sample posts (3 exist, all non-draft)

| File | Date | Slug |
|------|------|------|
| `hello-world.mdx` | 2026-04-13 | hello-world |
| `building-with-collections.mdx` | 2026-04-10 | building-with-collections |
| `themes-and-customization.mdx` | 2026-04-07 | themes-and-customization |

Sorted by date descending: hello-world (newest) -> building-with-collections -> themes-and-customization (oldest).

Per acceptance criteria: "Previous" = older (earlier date), "Next" = newer (later date). So for `building-with-collections`: prev = `themes-and-customization`, next = `hello-world`.

### CSS custom properties available

From `docs/CSS-ARCHITECTURE.md` and `src/styles/global.css`:
- `--color-text-muted` for secondary text
- `--color-border` for the separator line
- `--color-accent` / `--color-accent-hover` for link styling
- `--space-*` for spacing
- `--text-sm` for smaller text
- `--radius` for border radius
- `--transition` for hover transitions

### Layout structure

`[slug].astro` renders inside `Content.astro`, which wraps content in `<div class="page__content prose">`. The nav should go **after** `<RenderedContent />` but still inside the `<Content>` slot, so it sits at the bottom of the article content area.

### Architectural fit

- This is a **framework file** change (not site-specific). The nav must work generically for any Loomwork site that has posts.
- The `getStaticPaths()` function already fetches all posts. We need a second `getCollection()` call in the component script (outside `getStaticPaths`) to get the sorted list and find neighbors, since `getStaticPaths` only passes the single post in props.
- Alternatively, we can compute prev/next inside `getStaticPaths()` and pass them as additional props. This is **more efficient** (one collection fetch, computed at route generation time) and is the preferred approach.


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Approach:** Compute `prevPost` and `nextPost` inside `getStaticPaths()` and pass them as props. Render a `<nav>` element after `<RenderedContent />` within the `<Content>` layout slot.

**Why this approach over alternatives:**

1. **Props from getStaticPaths (chosen):** Single collection fetch, prev/next computed at build time during route generation. No redundant queries. Clean data flow.
2. *Alternative: Second getCollection() call in component script.* Would work but fetches the collection twice (once in getStaticPaths, once in the page component). Wasteful.
3. *Alternative: Separate component file.* Overkill for a simple nav bar that's only used in one place.

**Sort order:** Date descending (newest first), matching the listing page (`posts/index.astro` line 8). "Previous" = older post (index + 1 in descending array), "Next" = newer post (index - 1).

**Tradeoffs:**
- Adds ~40 lines of template + ~50 lines of scoped CSS to `[slug].astro`. Single-file change, no new components.
- The nav is inside the `<Content>` slot, so it inherits the content container width. This is correct behavior.


## Implementation Plan
<!-- Agent fills this during /ticket-investigate -->
- [ ] 1. **`src/pages/posts/[slug].astro` - Modify `getStaticPaths()`** (lines 5-11): Sort posts by date descending after fetching. For each post, find its index in the sorted array. Compute `prevPost` (index + 1, i.e. older) and `nextPost` (index - 1, i.e. newer), each as `{ title: string, slug: string } | null`. Pass both in the `props` object alongside `post`.
- [ ] 2. **`src/pages/posts/[slug].astro` - Destructure new props** (line 13): Update `const { post } = Astro.props;` to `const { post, prevPost, nextPost } = Astro.props;`.
- [ ] 3. **`src/pages/posts/[slug].astro` - Add nav markup** (after line 37, after `<RenderedContent />`): Add a `<nav class="post-nav" aria-label="Post navigation">` containing two links. Left side: if `prevPost`, render `<a href={"/posts/" + prevPost.slug} class="post-nav__link post-nav__link--prev">` with a left arrow (Unicode `\u2190`) and the previous post title. Right side: if `nextPost`, render the same pattern with a right arrow (`\u2192`). Use a simple flexbox layout with `justify-content: space-between`.
- [ ] 4. **`src/pages/posts/[slug].astro` - Add scoped CSS** (after existing `</style>` content, inside the `<style>` block): Style `.post-nav` as a flex container with a top border (`1px solid var(--color-border)`), padding-top (`var(--space-xl)`), margin-top (`var(--space-xl)`). Style `.post-nav__link` with `color: var(--color-accent)`, `text-decoration: none`, `font-size: var(--text-sm)`, hover state using `var(--color-accent-hover)`. Style `.post-nav__link--prev` to align left, `.post-nav__link--next` to align right with `margin-inline-start: auto`. Add responsive adjustments for small screens (stack vertically below 480px).
- [ ] 5. **Verify no em dashes** in any added copy (acceptance criterion). Use arrow characters only.
- [ ] 6. **Run `npm run build`** to confirm zero errors.
- [ ] 7. **Run `npx astro check`** to confirm type safety.
